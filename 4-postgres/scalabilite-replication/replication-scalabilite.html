La scalabilité et la réplication sont des aspects cruciaux pour garantir les performances et la fiabilité des bases de données PostgreSQL. Voici un aperçu complet de ces concepts :

1. Scalabilité dans PostgreSQL

1.1 Scalabilité Verticale

La scalabilité verticale consiste à améliorer les performances en ajoutant des ressources matérielles à un seul serveur PostgreSQL, telles que :

  • Processeurs (CPU) : Ajouter plus de cœurs ou des processeurs plus rapides.
  • Mémoire (RAM) : Augmenter la mémoire pour améliorer les performances du cache.
  • Stockage : Utiliser des disques SSD pour des I/O plus rapides.

Optimisation des Paramètres de PostgreSQL

Pour tirer le meilleur parti de la scalabilité verticale, vous pouvez ajuster plusieurs paramètres de configuration :

  • shared_buffers : La quantité de mémoire allouée aux tampons partagés. Représente généralement 25% de la mémoire totale.

    shared_buffers = 4GB

  • work_mem : La mémoire allouée pour les opérations internes de tri et de hachage.

    work_mem = 64MB

  • maintenance_work_mem : La mémoire utilisée pour les opérations de maintenance comme VACUUM, CREATE INDEX, etc.

    maintenance_work_mem = 512MB

  • effective_cache_size : Une estimation de la quantité de mémoire disponible pour le cache de disque du système d'exploitation.

    effective_cache_size = 12GB

1.2 Scalabilité Horizontale

La scalabilité horizontale consiste à distribuer la charge sur plusieurs serveurs. Dans PostgreSQL, cela peut être réalisé via :

  • Répartition des données (Sharding) : Partitionner les données et les distribuer sur plusieurs serveurs.
  • Réplication : Répliquer les données sur plusieurs serveurs pour équilibrer la charge de lecture.

2. Réplication dans PostgreSQL

La réplication permet de copier des données d'une base de données PostgreSQL (serveur maître) vers une ou plusieurs bases de données (serveurs esclaves).

2.1 Réplication Asynchrone

Dans la réplication asynchrone, le serveur maître envoie les changements aux serveurs esclaves sans attendre de confirmation. Cela améliore les performances, mais peut entraîner une légère désynchronisation.

Configuration de la Réplication Asynchrone

  1. Configurer le Serveur Maître

    Modifiez postgresql.conf :

    wal_level = replica
max_wal_senders = 5
wal_keep_segments = 32

    Modifiez pg_hba.conf pour permettre la connexion des esclaves :

    host replication all 0.0.0.0/0 md5

    Redémarrez PostgreSQL.

    Créez un utilisateur pour la réplication :

    CREATE USER replicator REPLICATION LOGIN ENCRYPTED PASSWORD 'password';

  2. Configurer le Serveur Esclave

    Clonez la base de données maître :

    pg_basebackup -h master_ip -D /var/lib/postgresql/data -U replicator -v -P

    Créez un fichier recovery.conf dans le répertoire de données de l'esclave :

    standby_mode = 'on'
primary_conninfo = 'host=master_ip port=5432 user=replicator password=password'
trigger_file = '/tmp/trigger_file'

    Redémarrez PostgreSQL sur l'esclave.

2.2 Réplication Synchrone

Dans la réplication synchrone, le serveur maître attend une confirmation de l'esclave avant de valider une transaction. Cela assure une haute disponibilité des données, mais peut affecter les performances.

Configuration de la Réplication Synchrone

  1. Configurer le Serveur Maître

    Modifiez postgresql.conf :

    synchronous_commit = on
synchronous_standby_names = '*'

  2. Configurer le Serveur Esclave

    La configuration est similaire à celle de la réplication asynchrone, sauf que synchronous_commit doit être activé sur le maître.

2.3 Réplication en Cascade

La réplication en cascade permet à un serveur esclave de répliquer ses données à d'autres esclaves. Cela réduit la charge sur le maître.

Configuration de la Réplication en Cascade

  1. Configurer le Serveur Maître

    Modifiez postgresql.conf :

    wal_level = replica
max_wal_senders = 10

  2. Configurer le Serveur Esclave (Niveau 1)

    Clonez la base de données maître et configurez comme précédemment.

    Ajoutez wal_level et max_wal_senders pour permettre la réplication en cascade :

    wal_level = replica
max_wal_senders = 5

  3. Configurer le Serveur Esclave (Niveau 2)

    Clonez la base de données de l'esclave de niveau 1 et configurez recovery.conf pour se connecter à l'esclave de niveau 1.

2.4 Réplication Logique

La réplication logique permet de répliquer des sous-ensembles de données et de schémas de base de données spécifiques, ce qui est utile pour les migrations et l'intégration de données.

Configuration de la Réplication Logique

  1. Configurer le Serveur Maître

    Modifiez postgresql.conf :

    wal_level = logical
max_replication_slots = 4
max_wal_senders = 4

    Créez un slot de réplication :

    SELECT * FROM pg_create_logical_replication_slot('my_slot', 'pgoutput');

  2. Configurer le Serveur Abonné

    Créez une souscription sur le serveur abonné :

    CREATE SUBSCRIPTION my_subscription CONNECTION 'host=master_ip dbname=mydb user=replicator password=password' PUBLICATION my_publication;

  3. Publier les Tables sur le Serveur Maître

    Créez une publication sur le serveur maître :

    CREATE PUBLICATION my_publication FOR TABLE my_table;

Pour plus de détails, vous pouvez consulter la documentation officielle de PostgreSQL.
